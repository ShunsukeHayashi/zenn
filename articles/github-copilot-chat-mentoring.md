こんにちは！フリーランスエンジニア歴7年のハヤシシュンスケです。

突然ですが、新人エンジニアへのメンタリングで消耗していませんか？

私が初めてメンターを任されたのは3年前のことです。当時は「教えることも仕事のうち」と思っていましたが、いざやってみると想像以上にきつかった。毎日30〜40分の質問対応、PRレビューごとに同じことを何度も説明、そして本業のコーディングが後回しになる日々。チームに新人が2人加わった月は、自分のタスクが全然消化できなくて、正直「自分に向いていないのかも」とまで思いました。

でも半年前、GitHub Copilot Chatをメンタリングに本格活用し始めてから、状況が劇的に変わりました。**メンタリングに使う時間が3分の1に減り**、なのに新人の成長スピードは上がり、**PRレビューでの指摘件数が80%減**った。今日はその実体験をシェアします。

## 😰 メンタリングの現実：よくある壁

「コードレビューするたびに同じコメントを書いている」「エラーメッセージを見せられても、まず文章で説明するところから始まる」「設計の意図を伝えるのに30分かかる」——こういう経験、ありませんか？

私が抱えていた主な課題はこんな感じでした。

- **非同期で質問に答えられない**: Slackでの質問は夜中に来ることもあって、返答が翌朝になる
- **同じ説明の繰り返し**: `undefined is not a function` 系エラーの説明を何度したか数えられない
- **文脈の共有コスト**: 「なぜこの設計にしたか」を毎回ゼロから伝える必要がある

これらがGitHub Copilot Chatで一気に解決されました。順番に説明します。

## 🚀 GitHub Copilot Chatとの出会い・衝撃

最初のきっかけは、新人のKくんが「このTypeScriptのエラー、意味がわからないです」とSlackで送ってきたときです。

```
Type 'string | undefined' is not assignable to type 'string'.
  Type 'undefined' is not assignable to type 'string'.
```

いつもなら私が「Optional chainingを使うか、型ガードを入れてください。理由は...」と文章を書く流れ。でもそのとき試しに「Copilot Chatに聞いてみて」と返したんです。

Kくんから数分後に「解決しました！！しかも理由まで教えてもらえました！！」と来た。

それだけじゃなかった。彼が「なんでこのエラーが出るのか、Copilotが図で説明してくれたんですけど、本当にわかりやすかったです」とも言っていた。

私が伝えるより**丁寧で、しかも即時**だった。これは使えると確信した瞬間でした。

## ⚠️ 3つの大失敗：AIメンタリングの落とし穴

ただ、最初から順調だったわけじゃありません。盛大にやらかしました。

### 失敗1: AIに頼りすぎて理解が浅くなった

「Copilot Chatに聞いてください」と言い続けた結果、新人のMさんが「動きましたが、なぜ動くのかわからないです」という状態に。

コードは書けるようになるけど、概念が身についていない。自分で考えずにAIのコードをコピペするようになってしまった。

**学んだこと**: 「Copilotに聞いて → 自分の言葉で説明してみて → それを私に教えて」の3ステップが必須。AIは答えを教えてくれるが、理解を保証しない。

### 失敗2: コンテキストなしで質問させていた

「エラーが出ます」「動きません」みたいな曖昧な質問をそのままCopilot Chatに投げさせていた。当然、回答もズレる。

新人はそもそも「何を伝えるべきか」を知らないことが多い。「どんな状況でエラーが起きているか」「自分が試したこと」「期待する動作」を含めて質問する方法を教えていなかった。

**学んだこと**: 質問の仕方そのものをトレーニングする必要がある。良い質問テンプレートを作って渡した。

```
# 質問テンプレート（新人向け）
## 状況
[やろうとしていること]

## 発生しているエラーまたは問題
[エラーメッセージや具体的な症状]

## 試したこと
[すでに試した対処法]

## コード
[関連するコードスニペット]
```

### 失敗3: メンター不要論を信じかけた

効率化が進むにつれて「もうCopilotがいれば私は必要ないのでは？」という考えが頭をよぎった。

でもそれは大きな間違いだった。AIは個人の成長曲線を把握できない。チームの雰囲気や暗黙知を伝えられない。「なぜこのチームではこの判断をするのか」というコンテキストはAIには提供できない。

**学んだこと**: Copilot Chatは「情報提供」が得意で、「判断と文化の継承」は人間にしかできない。役割分担が大事。

## 📊 劇的な数値改善 Before/After

半年間の変化を正直にまとめます。

| 指標 | Before | After | 改善率 |
|------|--------|-------|--------|
| 1日あたりの質問対応時間 | 約40分 | 約13分 | -68% |
| PRレビューの指摘件数（週平均） | 約25件 | 約5件 | -80% |
| 新人が自己解決した割合 | 約30% | 約75% | +150% |
| メンターとしての満足度（主観） | 3/10 | 8/10 | — |

特に印象的だったのは「自己解決率」の変化です。以前は「わからない→メンターに聞く」がデフォルトだったのが、今は「わからない→Copilotに聞く→理解できた→メンターに確認」という流れになった。私の出番が「確認」に変わったことで、質問の質が格段に上がりました。

## 🛠️ 実践手順5ステップ

では具体的にどう使うか、ステップ別に解説します。

### ステップ1: コードレビューの説明を自動化する

PRレビューでコメントを書く前に、まずCopilot Chatで説明文を生成してもらいます。

```
# Copilot Chatへのプロンプト例
以下のコードの問題点を、入門者向けにわかりやすく説明してください。
なぜ問題なのか、どう直すべきか、そしてその理由も含めて教えてください。

[問題のあるコードを貼り付け]
```

これで生成された説明文を元に、自分の言葉でコメントを書き直す。最初は「そのまま使えばいいじゃん」と思いましたが、自分の言葉に変換することでチームの文脈に合った説明になります。

### ステップ2: エラーメッセージの解説を委任する

エラーが出たらまず新人にこう伝えます。「そのエラーメッセージと、エラーが起きているコードをCopilot Chatに貼り付けて、説明してもらってください」

その後、「理解できた？自分の言葉で説明して」と聞く。ここが肝心です。

```typescript
// こういうエラーが出たとき
const user = getUser(); // 戻り値が User | undefined
console.log(user.name); // Object is possibly 'undefined'.
```

Copilot Chatは`undefined`チェックの必要性から、Optional Chaining（`?.`）やNullish Coalescing（`??`）の使い方まで丁寧に説明してくれます。私が説明するより充実していることも多い。

### ステップ3: 設計パターンの提案を視覚化する

「なぜこの設計にしたか」を伝えるのが一番時間のかかる部分でした。これをCopilot Chatに図示してもらいます。

```
# プロンプト例
Repositoryパターンを使う理由を、直接DBアクセスとの比較を含めて
初心者向けに説明してください。メリット・デメリットも含めてください。
```

私が口頭で30分かけて説明していた内容が、2分で新人の手元に届く。そして理解したかどうかを確認する時間に残り28分を使える。

### ステップ4: テストコードの教え方を変える

テストの書き方は特に教えるのが難しかった。「なぜテストを書くのか」という概念から、「どう書くのか」という実装まで、同時に伝えないといけないから。

```typescript
// 新人が書いたコード
function calculateTax(price: number): number {
  return price * 0.1;
}

// Copilot Chatへのプロンプト
// 「この関数のユニットテストを書いてください。
//  また、なぜこのテストケースが必要なのかも説明してください」
```

Copilot Chatはテストコードと一緒に「なぜこのケースをテストするのか」も説明してくれます。新人は実装とその意図をセットで学べる。

### ステップ5: リファクタリングのガイドを実装と並走させる

「このコードを改善してください」よりも「このコードの問題点を教えて、改善の方向性を示してください」という使い方をします。

```typescript
// リファクタリング前（新人が書いたコード）
async function fetchUserAndOrder(userId: string) {
  const userRes = await fetch(`/api/users/${userId}`);
  const user = await userRes.json();
  const orderRes = await fetch(`/api/orders?userId=${userId}`);
  const orders = await orderRes.json();
  if (orders.length > 0) {
    for (let i = 0; i < orders.length; i++) {
      console.log(orders[i].id);
    }
  }
  return { user, orders };
}
```

```
# Copilot Chatへのプロンプト
このコードの問題点を3つ挙げてください。
それぞれについて「なぜ問題なのか」「どう直すべきか」を教えてください。
一度に全部直すのではなく、優先順位をつけて段階的に改善する方法も提案してください。
```

「全部直してください」ではなく「問題点を教えて、段階的に改善する」ことで、新人が「リファクタリングとは何か」を体験的に学べます。

## 🌐 応用パターン：チーム全体への展開

個人のメンタリングで効果が出たら、チームに広げましょう。

**チーム共有プロンプト集の作成**

私が効果的だったプロンプトをNotionにまとめて共有しています。「TypeScriptの型エラーを解説するプロンプト」「PRレビューコメントの説明を生成するプロンプト」などのテンプレート集です。

**ペアプログラミングへの組み込み**

ペアプロ中にCopilot Chatを第3の参加者として扱う。「Copilotに聞いてみよう」「この判断についてCopilotはどう言う？」という使い方で、議論が活発になります。

**週次の振り返りセッション**

週に一度、「今週Copilot Chatに質問したこと」を共有する時間を設けました。チーム全体で良いプロンプトの使い方を学び合うことができます。

**注意点**: Copilot Chatの回答を鵜呑みにしないこと。特に古いライブラリのバージョンに関する情報や、セキュリティ関連のコードは必ず人間がレビューしてください。AIは自信満々に間違えることがあります。

## まとめ：AIはメンターを置き換えない、拡張する

GitHub Copilot Chatを使い始めて半年、私が感じた一番大きな変化は「メンタリングが嫌いじゃなくなった」ことです。

単純な情報伝達の作業がAIに委任されて、私は「なぜそのキャリアを選んだか」「チームでどう働くか」「困ったときにどう考えるか」という、人間にしか伝えられない部分に集中できるようになりました。

**メンタリング時間3倍効率**というのは、同じ時間でより深い関わりができるということ。**PRレビュー指摘80%減**というのは、新人が自分で考えて書けるようになったということ。

AIはメンターを不要にするのではなく、より良いメンターになるための道具です。

ぜひ試してみてください。最初の一歩は「次に新人から質問が来たら、まずCopilot Chatに聞かせてみる」だけでOKです。
