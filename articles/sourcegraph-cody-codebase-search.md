フリーランスエンジニア7年目のハヤシです。

今日は「巨大なコードベースに放り込まれたとき、どう生き延びるか」という話をします。

先日、あるSaaS企業の技術顧問として参画したとき、初日に「まずソースコードを読んでおいてください」と言われました。Gitリポジトリをcloneしてみると、**ファイル数12,000超、コミット数50,000超**のモノレポが鎮座していました。

`grep -r "UserService"` を実行したら5分待っても終わりませんでした。

その体験から3ヶ月後、Sourcegraph Codyを使いこなした今では、**同じ規模のコードベースで「この機能どこで実装されてる？」という問いに30秒で答えられるようになりました。**

---

## Sourcegraph Codyとの出会い

Sourcegraphは元々コード検索ツールとして有名でしたが、**Cody**はそのAIアシスタント機能です。単なるコード補完ではなく、「コードベース全体を文脈として持つAI」という点が他のツールと一線を画しています。

GitHub Copilotは開いているファイルとその周辺しか見ません。ChatGPTはコードを貼り付けないと何も知りません。でもCodyは、**リポジトリ全体をインデックスして、「このプロジェクト全体の文脈」でAIが答えてくれます。**

最初に知ったのはTwitter（X）で流れてきたデモ動画でした。「15万行のコードベースで `この機能のバグはどこ？` と聞いたら、ファイルとコンテキストを返してきた」というやつです。半信半疑で試したのが始まりでした。

## フリーランス7年で踏んだ失敗談3選

Codyを語る前に、私がいかにコード検索で苦しんできたかを正直に話します。

### 失敗1：`grep` 職人になりすぎた

フリーランス1年目、とにかく `grep` を使いまくっていました。

```bash
# 当時の私の「最強コード検索」
grep -r "payment" . --include="*.ts" | grep -v node_modules | grep -v ".test."
```

これで見つかったファイルを手動で開いて、関連ファイルをまた `grep` して……。**1つの機能を理解するのに平気で2〜3時間かかっていました。**

クライアントには「調査中です」と言いながら、正直なところコードの迷宮で迷子になっていました。フリーランスとして時間単価をいただいているのに、その大半をコード解読に費やしている罪悪感がありました。

### 失敗2：コードベースのコンテキストなしにAIに質問した

ChatGPT・Claude・GitHub Copilotが普及してから、「AIに聞けばいい」という雑な使い方をしていた時期があります。

「このエラーを直して」とコードを貼り付けても、プロジェクト固有の命名規則、独自のユーティリティ関数、DBスキーマの構造——そういった「文脈」を毎回手動で補足しなければなりませんでした。

あるとき、認証まわりのバグ修正を頼まれて、関連する可能性のあるファイルを5つ貼り付けてClaudeに聞きました。返ってきた回答は技術的には正しいものでしたが、**プロジェクトで使っているカスタムの `AuthMiddleware` を無視した提案で、そのまま実装したら別のバグが生まれました。** クライアントへの謝罪と修正で半日消えました。

### 失敗3：大規模リファクタリングで影響範囲を読み誤った

フリーランス5年目、中規模のECサイトで「決済モジュールのリファクタリング」を担当しました。`PaymentService` クラスを綺麗にしようとして変更を加えたところ、**思わぬ10箇所でimportが壊れていました。**

テストは通っていたのに、なぜか本番デプロイ後に管理画面の返金機能だけが動かなくなりました。調査して判明したのは、管理画面側のコードが `PaymentService` の内部メソッドを直接呼び出していたこと。そのファイルはテストカバレッジがほぼゼロでした。

影響範囲の把握が甘かった。それだけです。**リファクタリング前に「このクラスを参照しているコードを全部洗い出す」という作業を、当時の私は完璧にはできていませんでした。**

3ヶ月間、いくつかのプロジェクトで意識的に計測しました。

|---|---|---|

## 実践5ステップ：Codyを使いこなす

### Step 1：セットアップ（15分でできる）

まずVS Codeに拡張機能をインストールします。

```bash
# VS Code拡張機能のインストール（コマンドパレットから）
# または以下のIDで検索
# sourcegraph.cody-ai
```

インストール後、Sourcegraphのアカウントを作成してサインインします。無料プランでも個人利用には十分です。

**ポイント：** 大規模コードベースを扱うなら、まずリポジトリのインデックスを作成します。Sourcegraph.comに接続してリポジトリを追加するか、セルフホスト版を使います。VS Codeのサイドバーに「Cody」が表示されれば準備完了です。

### Step 2：コード検索をCodyに任せる

これが最初の大きな変化です。`grep` の代わりにCodyのチャットで聞きます。

**使用例：**

```
# チャットに入力する質問の例
「認証（authentication）の処理はどこで行われていますか？
ユーザーがログインしてからJWTトークンが発行されるまでの流れを教えてください」
```

Codyはコードベースをインデックスから検索し、関連ファイルと処理の流れを返してくれます。単にファイルパスを返すだけでなく、**「なぜそこを見るべきか」という文脈も説明してくれます。**

### Step 3：AIへの質問をコンテキスト付きで行う

Codyの真価は「コードベース全体を文脈として持つ」点です。チャットで右クリックして「Add to Cody」でコンテキストにファイルを追加できますが、普通に質問するだけでもインデックスから関連コードを引っ張ってきます。

```
# 実際の活用例
「src/services/UserService.ts の deleteUser メソッドを修正したいのですが、
このメソッドを呼び出している箇所と、削除時に関連して処理が必要な
テーブルを教えてください」
```

これだけで、呼び出し箇所の一覧と、外部キー制約が絡むテーブルまで教えてくれます。**以前なら30分かかっていた調査が5分で終わります。**

### Step 4：リファクタリングの影響範囲を事前調査

前述の失敗談の教訓から、リファクタリング前に必ずCodyで影響範囲を確認するようにしました。

```
# リファクタリング前の確認プロンプト
「PaymentService クラスの processPayment メソッドのシグネチャを変更しようとしています。
（引数に `currency: string` を追加します）
このメソッドを直接・間接的に呼び出しているファイルをすべて教えてください。
また、この変更によって影響を受けそうな処理を洗い出してください」
```

Codyが関連ファイルを列挙してくれるので、変更前にすべてのファイルを確認できます。**「まさかここまで影響するとは…」という事故がほぼなくなりました。**

### Step 5：新規機能の実装パターンを既存コードから学ぶ

新機能を実装するとき、「このプロジェクトのコーディングスタイルに合わせて書く」のが大事です。Codyはそれを助けてくれます。

```
# 実装パターン確認のプロンプト
「このプロジェクトでは新しいAPIエンドポイントをどのように追加していますか？
既存の実装例（例：ユーザー関連のエンドポイント）を参考に、
/api/v1/orders の GET エンドポイントを追加する手順を教えてください。
エラーハンドリングや認証ミドルウェアの適用方法も含めて」
```

プロジェクト固有の `BaseController` の使い方、エラーレスポンスの形式、バリデーションの書き方——こういった「暗黙のルール」をCodyが既存コードから読み取って教えてくれます。

## セキュリティ・プライバシーについて

フリーランスで複数クライアントのコードを扱う場合、当然「コードがどこに送られるか」は気になります。

Codyはコードをトレーニングデータに使用しない設定が可能で、エンタープライズプランではオンプレミス（セルフホスト）対応もあります。クライアントのコードを扱う場合は、必ずCodyのプライバシーポリシーを確認した上で、クライアントへの説明と承認を得ることを推奨します。

私の場合、機密性の高いプロジェクトではセルフホスト版のSourcegraphを建てて使っています。

## GitHub Issueとの組み合わせ

Codyで影響範囲を調査した後は、GitHub Issueに変更計画を記録しています。「このIssueで何を変更するか」「影響ファイルはどこか」をIssueコメントに書いておくと、後のコードレビューが格段にスムーズになります。

GitHub IssueとAIを組み合わせた自動化については[GitHub Issues × AI自動化で開発フローを変える](/articles/github-issues-ai-automation)で詳しく解説しています。

## まとめ：巨大コードベースは怖くない

Sourcegraph Codyを使い始めてから、「知らないコードベースに入る」ことへの恐怖がほぼなくなりました。

- 全体像の把握：Codyに「このプロジェクトのアーキテクチャを教えて」と聞く
- 機能の場所特定：自然言語で「〇〇の処理はどこ？」と聞く
- 影響範囲の確認：変更前にCodyで依存関係を洗い出す
- 実装パターンの踏襲：「既存コードに倣って〇〇を実装して」と依頼する

フリーランスとして新規参画のたびに感じていた「コードベース解読コスト」は、Codyによって劇的に削減できます。7年間 `grep` と格闘してきた私が言うので、間違いありません。

まずは自分の既存プロジェクトで試してみてください。きっと「もっと早く知りたかった」と思うはずです。
