## Jiraに疲れたあなたへ

「スプリント計画に3時間かかる」「チケット更新を忘れたら詰められる」「ロードマップが常に古い」——フリーランスエンジニア7年目の自分が、Jiraを使い続けた5年間の正直な感想がそれだった。

プロジェクト管理ツールそのものが、プロジェクトの足を引っ張っている。

そんな状況が変わったのは、LinearとAIを組み合わせた運用を始めてから。計画精度が55%から90%に跳ね上がり、週次の管理作業が8時間から2時間に縮んだ。本記事では、その実践記録を失敗談も含めて正直に書く。

---

## Linearを初めて触ったときの衝撃

Linearとの出会いは、あるスタートアップ案件でクライアントから「うちはLinear使ってるので合わせてほしい」と言われたことがきっかけだった。

最初の印象は「Jiraのミニマル版か」くらいだったが、30分使って認識が変わった。

**速さが違う。** イシュー作成からステータス変更まで、すべての操作がJiraの3分の1以下の時間で終わる。キーボードショートカットが全操作に割り当てられており、マウスがほぼ不要。バックログの整理やスプリント組み替えが、まるでテキストエディタでコードを書くような感覚でできる。

そしてLinear AIの存在を知ったとき、さらに驚いた。

イシューのタイトルを入力するだけで、説明文・受け入れ条件・サブタスクの叩き台を自動生成してくれる。チームのコーディングパターンや過去イシューをコンテキストにした推定工数も出してくれる。「これをJiraで同じことをやったら何時間かかるか」と考えると、鳥肌が立った。

---

## やらかした失敗談3つ

順調に見えるかもしれないが、導入初期に盛大に失敗した。同じ轍を踏まないように正直に書いておく。

### 失敗1：AI生成チケットをそのまま信じた

Linear AIが生成するイシュー説明は精度が高い。高いのだが、最初の2ヶ月は「AIが書いたんだから正しいだろう」という思考停止に陥った。

具体的には、APIエンドポイント追加のイシューでAIが生成したサブタスクに「認証ミドルウェアの実装」が含まれていた。実際には認証はすでに共通化されていたので不要な作業だった。それに気づかないまま着手し、2日間溶かした。

**学んだこと：** AI生成の内容は「叩き台」として扱う。必ずコードベースの実態と照合してから承認する。

### 失敗2：AI見積もりを序盤から本番投入した

Linearの工数推定AIは、チームの過去データが蓄積されるほど精度が上がる仕組みだ。ところが導入直後、データが十分でない段階から「AIが言った工数でクライアントに見積もりを出す」運用を始めてしまった。

3ヶ月目に大型機能の見積もりを出したとき、AIの推定は2週間。実際は4週間かかった。クライアントとの信頼に傷がつき、調整に丸一日費やした。

**学んだこと：** AI見積もりの本番投入は、チームの過去イシューが最低50件蓄積されてから。それまでは参考値として使い、最終判断は人間が行う。

### 失敗3：チームへの根回しゼロで導入した

これが最大の失敗だった。「LinearはJiraより絶対良い。切り替えれば全員幸せになる」と確信していた自分は、チームミーティングで5分説明しただけで切り替えを強行した。

案の定、Jiraに慣れたベテランエンジニアから強い抵抗を受けた。「UIが違いすぎる」「Jiraとの連携が切れる」「学習コストを誰が払うのか」——すべてもっともな指摘だった。

移行後1ヶ月間、チームの生産性はむしろ下がった。

**学んだこと：** ツール移行は技術的な問題ではなく、人の問題。2週間かけて全員と1on1で懸念を聞き、段階的移行プランを作ってから実施すべきだった。

## 改善数値：Before/After

失敗を経て運用を安定させた後の数値がこちら。フリーランスとして並行稼働している3プロジェクトの平均値。

| 指標 | 導入前（Jira） | 導入後（Linear+AI） |
|------|--------------|-------------------|
| 週次管理作業時間 | 8時間 | 2時間 |
| スプリント計画精度 | 55% | 90% |
| イシュー作成時間（1件） | 8分 | 2分 |
| バックログ整理頻度 | 隔週 | 随時（5分/回） |
| 見積もり乖離率 | 平均35%超過 | 平均8%以内 |

計画精度90%というのは「スプリントで計画したポイントの90%以上を完了できた回の割合」を指す。Jira時代は半分以上のスプリントが消化不良で終わっていた。

## 実践5ステップ

理論ではなく、今すぐ再現できる手順を書く。

### ステップ1：Linearの初期セットアップ

プロジェクトを作成したら、まずワークフローのカスタマイズから入る。デフォルトの`Todo / In Progress / Done`は最低限でいいが、自分のチームには`Review / Blocked`を追加した。

ラベルはシンプルに保つ。`bug`, `feature`, `chore`, `tech-debt`の4つで十分。Jira時代に20個以上のラベルを作って破綻した経験から学んだ。

チームメンバーを招待したら、最初の1週間は既存のJiraチケットをLinearにインポートせず、新規イシューだけLinearで管理する並行運用を推奨する。急な切り替えによる抵抗を最小化できる。

### ステップ2：Linear AIでイシュー作成を高速化

Linear AIを有効にすると、イシュータイトルを入力した段階でAIが説明文の提案を始める。

効果的な使い方は「タイトルに動詞と対象を明示すること」。

- 悪い例：`ログイン改修`
- 良い例：`ログインフォームにパスワード強度インジケーターを追加する`

後者のほうが、AIが正確な受け入れ条件とサブタスクを生成しやすい。生成されたサブタスクは必ず一度読んで、不要なものを削除してから保存する。

### ステップ3：AIによるスプリント計画

スプリント計画のMTGの前日、Linearの`Insights`から前スプリントのベロシティデータをエクスポートする。これをClaude（またはChatGPT）に渡し、次のプロンプトで計画の叩き台を作らせる。

```
以下はわれわれのチームの直近5スプリントのデータです。
[エクスポートデータを貼り付け]

次のスプリント（2週間）の候補イシューリストも添付します。
[バックログのイシュー一覧を貼り付け]

条件：
- チームキャパシティは前スプリント平均の80%を上限とする
- P0バグは必ず含める
- 技術的負債を20%確保する

最適なスプリント構成を提案し、理由も説明してください。
```

このAI提案をMTGの叩き台にすることで、スプリント計画の所要時間が3時間から45分に短縮された。

### ステップ4：GitHub ActionsとLinearのCI連携

コミットメッセージやPRタイトルにLinearのイシューIDを含めることで、開発状況がリアルタイムで反映される。

```yaml
# .github/workflows/linear-sync.yml
name: Linear Issue Sync

on:
  pull_request:
    types: [opened, closed, reopened]

jobs:
  sync-linear:
    runs-on: ubuntu-latest
    steps:
      - name: Extract Linear Issue ID
        id: extract
        run: |
          # PRタイトルまたはブランチ名からLinear IDを抽出
          # 例: feat/ENG-123-add-login-indicator
          ISSUE_ID=$(echo "${{ github.head_ref }}" | grep -oP '[A-Z]+-\d+' | head -1)
          echo "issue_id=$ISSUE_ID" >> $GITHUB_OUTPUT

      - name: Update Linear Issue Status
        if: steps.extract.outputs.issue_id != ''
        run: |
          # PRオープン時はIn Reviewへ、マージ時はDoneへ遷移
          if [ "${{ github.event.action }}" = "closed" ] && [ "${{ github.event.pull_request.merged }}" = "true" ]; then
            STATUS="Done"
          else
            STATUS="In Review"
          fi

          curl -s -X POST https://api.linear.app/graphql \
            -H "Authorization: ${{ secrets.LINEAR_API_KEY }}" \
            -H "Content-Type: application/json" \
            -d "{
              \"query\": \"mutation { issueUpdate(id: \\\"${{ steps.extract.outputs.issue_id }}\\\", input: { stateId: \\\"$STATUS\\\" }) { success } }\"
            }"
```

このワークフローにより、PRを出すだけでLinearのステータスが自動更新される。手動更新忘れによるステータス乖離がゼロになった。

AIを使ったGitHub ActionsのCI/CD自動化の詳細は「[GitHub Issuesとの連携自動化](github-issues-ai-automation)」も参照してほしい。

### ステップ5：週次レビューをAIで自動化

毎週金曜日、LinearのAPIからその週のイシューデータを取得してAIに渡し、週次レポートを自動生成している。

```javascript
// weekly-report.js（Node.js）
const { LinearClient } = require("@linear/sdk");

async function generateWeeklyReport() {
  const client = new LinearClient({ apiKey: process.env.LINEAR_API_KEY });

  // 今週完了したイシューを取得
  const oneWeekAgo = new Date();
  oneWeekAgo.setDate(oneWeekAgo.getDate() - 7);

  const completedIssues = await client.issues({
    filter: {
      completedAt: { gte: oneWeekAgo.toISOString() },
      team: { key: { eq: "ENG" } },
    },
  });

  // イシューデータを整形してAIへ送信
  const issuesSummary = completedIssues.nodes.map((issue) => ({
    title: issue.title,
    estimate: issue.estimate,
    labels: issue.labels?.nodes?.map((l) => l.name),
    cycle: issue.cycle?.name,
  }));

  // Claude APIへ送信してレポート生成（省略）
  console.log("今週の完了イシュー:", JSON.stringify(issuesSummary, null, 2));
}

generateWeeklyReport();
```

このスクリプトをGitHub ActionsのCron（毎週金曜17:00）で走らせ、SlackにレポートをPOSTしている。毎週30分かけて書いていた週次報告がゼロになった。

## Jira / Notionとの比較

「結局どのツールが一番いいの？」という質問への答え：用途による。

| 項目 | Linear | Jira | Notion |
|------|--------|------|--------|
| 操作速度 | ◎ 非常に速い | △ 重い | ○ 普通 |
| AI機能 | ◎ 標準搭載 | △ 別途設定 | ○ AIブロック |
| カスタマイズ性 | ○ 必要十分 | ◎ 高い | ◎ 非常に高い |
| 学習コスト | ◎ 低い | △ 高い | ○ 中程度 |
| 価格（/人月） | $8〜 | $8.15〜 | $10〜 |
| GitHub連携 | ◎ ネイティブ | ○ プラグイン | △ 弱い |
| 向いているチーム | 小〜中規模開発 | 大規模・複雑PJ | ドキュメント主体 |

Jiraが輝くのは、コンプライアンス要件や複雑な承認フロー、100人超の大規模チームの場合。Notionはタスク管理よりもナレッジベースや仕様書管理として使ったほうが本領を発揮する。

Linearは「開発チームがコードを書くのと同じ感覚でプロジェクトを管理したい」というニーズにもっとも応える。

## まとめ

Linear×AI運用で変わった点を一言で言えば、「プロジェクト管理が思考を妨げなくなった」だ。

Jira時代は、管理作業そのものに認知負荷を取られていた。チケットを正しい階層に作るのに悩み、フィールドを埋めるのに時間を使い、会議でスライドを作るために別途情報を集めていた。

Linearに移行してからは、管理ツールの存在が透明になった。コードを書くことに集中できる時間が増え、計画の精度も上がり、クライアントへの信頼も高まった。

失敗談で書いたように、導入は万能ではないし、チームの合意なしには機能しない。でも適切に使えば、間違いなくエンジニアの時間と精神的コストを削減してくれるツールだ。

まずは個人プロジェクトや小規模な案件で試してみてほしい。最初のイシューを作った瞬間、その速さに驚くはずだ。
