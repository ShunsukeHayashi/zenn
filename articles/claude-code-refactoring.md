## 🚀 【衝撃の初体験】5000行のレガシーコードにClaudeを投入した結果

そのコードベースは2020年に外注で作られたECサイトのバックエンドでした。Node.jsで書かれていましたが、クラスの継承が深すぎて何を継承しているのか判別できず、データベースクエリは生のSQLが各所に散らばり、エラーハンドリングは`catch(e) { console.log(e) }`が至るところに...。

「全部書き直したい」という気持ちを抑えて、まずClaudeに現状分析を依頼しました。

```
このコードベースの技術的負債を分析してください。
特に以下の観点で問題点を洗い出してほしいです：
1. 保守性（命名規則、関数の複雑度）
2. 信頼性（エラーハンドリング、型安全性）
3. テスタビリティ（依存関係の注入、単体テスト可能性）
```

返ってきた分析レポートは45項目にわたる問題リストでした。しかも単に問題を列挙するだけでなく、「優先度High/Medium/Lowの3段階」と「修正工数の見積もり」まで付いていました。

**結果として3週間で主要な技術的負債を解消でき、当初想定していた3ヶ月の見積もりを75%短縮できました。** これが私のClaudeリファクタリング体験の始まりです。

## 😅 【3つの失敗談】やりがちなリファクタリングの罠

成功だけを話しても不誠実なので、正直に失敗談も話します。同じ轍を踏まないようにしてください。

### 失敗1：一気に全部やろうとして大炎上

最初の週、意気込んで「このファイル全体をきれいにして」とClaudeに投げました。400行のファイルが返ってきて、動作確認なしにそのままコミット。結果、本番でNullPointerExceptionが多発して2時間のダウンタイムを引き起こしました。

**教訓**: リファクタリングは必ず小さな単位で行い、各ステップでテストを実行する。一度に変更するのは50行以内に留める。

### 失敗2：Claudeの提案をそのまま使って型エラー地獄

「any型を全部解消して」と依頼したところ、Claudeが生成した型定義が既存のAPIレスポンス形式と微妙にずれていました。TypeScriptのコンパイルは通るのに、実行時に型変換エラーが出るという最悪のパターンに。

問題は私が「実際のAPIレスポンスのサンプルデータ」をClaudeに渡していなかったことでした。

**教訓**: 型リファクタリングをするときは必ず実際のデータサンプルをClaudeに提供する。`console.log`で出力したJSONを貼り付けるだけで精度が劇的に上がる。

### 失敗3：依存関係を無視して循環参照を生み出した

モジュール分割のリファクタリング中、ClaudeがAクラスがBクラスを参照し、BクラスがAクラスを参照する構造を提案してきました。一見クリーンに見えたのですが、循環参照でNode.jsが起動しなくなるという事態に。

**教訓**: 大きな構造変更をする前に、必ず「このリファクタリングで循環参照は発生しないか？」と明示的にClaudeに確認させる。

## 🛠️ 【実践手順】Claude Codeを使ったリファクタリングの完全ステップ

ここからが本題です。私が3週間で磨き上げた「再現性のあるリファクタリング手順」を公開します。

### STEP 1: コードベースの分析

まず全体像を把握します。Claude Codeのプロジェクト読み込み機能を使って、コードベース全体を解析させます。

```
以下の観点でこのコードベースを分析してください：

1. 技術的負債の洗い出し（重大度High/Medium/Lowで分類）
2. 各問題の修正工数見積もり（時間単位）
3. 修正する場合の依存関係（先に直すべきものはどれか）
4. リスクの高い変更はどれか

出力はMarkdownのテーブル形式でお願いします。
```

ポイントは「優先順位付き・依存関係付き」で出力してもらうこと。ランダムに直し始めると、後で依存関係の問題が出てきます。

### STEP 2: 問題箇所の特定

分析結果をもとに、まず「Highリスク・低工数」の問題から着手します。Quick Winを積み重ねることでモチベーションを維持できます。

各問題について、Claudeに詳細レポートを作成させます：

```typescript
// 例：このような any だらけの関数を発見した場合
async function processOrder(data: any): Promise<any> {
  const result = await db.query(`SELECT * FROM orders WHERE id = ${data.id}`);
  return result;
}
```

このコードをClaudeに見せて「何が問題か」を聞くと、SQLインジェクション、型安全性の欠如、エラーハンドリングの欠如という3つの問題を即座に指摘してくれます。

### STEP 3: リファクタリング計画

問題を特定したら、次は「どういう順番で直すか」を計画します。

**私が使っているプロンプトテンプレート：**

```
この関数のリファクタリング計画を作成してください。

制約条件：
- 一度に変更する行数は50行以内
- 各ステップで動作確認できる単位にする
- 既存のAPIインターフェースは変更しない（外部からの呼び出し方を変えない）
- テストを先に書いてからリファクタリングする（テストファースト）

期待する出力：
1. ステップ分割（番号付きリスト）
2. 各ステップの変更内容とリスク
3. 各ステップのテスト方法
```

### STEP 4: 段階的な実装

計画ができたら、一つひとつのステップを丁寧に実装します。私のルールは「1ステップ = 1コミット」。

先ほどの `processOrder` 関数なら、Claudeはこのように書き直してくれます：

```typescript
// リファクタリング後
type OrderId = string;

interface OrderProcessResult {
  orderId: OrderId;
  status: 'success' | 'failed';
  processedAt: Date;
}

async function processOrder(orderId: OrderId): Promise<OrderProcessResult> {
  // パラメータ化クエリでSQLインジェクション対策
  const result = await db.query(
    'SELECT * FROM orders WHERE id = $1',
    [orderId]
  );

  if (!result.rows.length) {
    throw new NotFoundError(`Order not found: ${orderId}`);
  }

  return {
    orderId,
    status: 'success',
    processedAt: new Date(),
  };
}
```

型安全、SQLインジェクション対策、適切なエラーハンドリングが揃った読みやすいコードに変わりました。

### STEP 5: テストとレビュー

最後のステップはテスト。Claudeにテストコードも書いてもらいます：

```
このリファクタリング後のコードに対するユニットテストをJestで書いてください。
以下のケースを必ずカバーしてください：
- 正常系：有効なorderIdで成功
- 異常系：存在しないorderIdでNotFoundError
- 異常系：DB接続エラーのハンドリング

モックはjest.mockを使用してください。
```

テストが通ったら、Claudeにコードレビューも依頼します。「このコードに問題はあるか？」と聞くと、自分では気づかなかった観点を指摘してくれることがあります。

## 🚀 【応用パターン】大規模プロジェクトへの適用

個人プロジェクトや小規模なコードベースで手順を掴んだら、大規模プロジェクトにも応用できます。

大規模プロジェクトでのポイントは「ドメイン境界に沿ったリファクタリング」です。いきなり全体を触るのではなく、「注文管理モジュールだけ」「ユーザー認証だけ」というように境界を明確にして、その範囲内で完結させます。

また、チームで作業する場合は、**リファクタリング専用のブランチ戦略**も重要です。私は「refactor/order-module」のような命名で、機能開発ブランチと完全に分離するようにしています。Claude Codeのworktree機能を活用すれば、メインの開発を止めずにリファクタリングを進められます。
