こんにちは！フリーランスエンジニア歴7年のハヤシシュンスケです。

「このコード、誰が書いたんだ...」と頭を抱えながら深夜にデバッグしていたあの日のことを、今でも鮮明に覚えています。10年前に外注で作られたPython製の在庫管理システム。コメントは一切なし、グローバル変数が至る所に散らばり、1つの関数が500行超え。それが40万行規模で存在する。そんな悪夢のようなコードベースと格闘することになったのが、この話の始まりです。

最初の見積りでは「全体の刷新に6ヶ月」と出ていました。それが**Aider×Claudeを導入することで2ヶ月に短縮**できた——今回はその実体験を包み隠さずお届けします。

---

## 🔥 40万行の「負債の塔」と向き合う

引き継いだシステムのスペックはこうでした。

- **言語**: Python 2系（！）から3.6に途中移行したが混在
- **行数**: 約40万行（テストコードゼロ）
- **依存関係**: `requirements.txt` が存在せず、環境構築に3日かかる
- **バージョン管理**: Git未使用。FTPサーバーに直接アップロードという運用
- **ドキュメント**: Wordファイルが4年前に更新停止

「リファクタリング」という言葉すら生ぬるい状態です。しかも、このシステムは今も現役で動いており、1日のダウンタイムが許されない業務システムでした。

手でやったら確実に1年以上かかる。そう判断した私がたどり着いたのが、**CLIベースのAIペアプログラマー「Aider」** との出会いでした。

## 💡 Aider×Claude の発見と衝撃

Aiderとは、ターミナルから使えるAIコーディングアシスタントです。Claude APIと組み合わせることで、巨大なコードベースに対してチャット形式でリファクタリング指示が出せます。

最初に試したのは、最も複雑な在庫計算モジュール（約3,000行）でした。

```bash
# Aider + Claude opus でレガシーコードを解析
aider --model claude-opus-4-5 --file legacy_inventory.py

# 複数ファイルをまとめて指定することも可能
aider --model claude-opus-4-5 \
  --file legacy_inventory.py \
  --file db_connector.py \
  --file utils.py
```

起動して最初に打ち込んだプロンプト：

```
このファイルの技術的負債を分析してください。
特に以下を重点的に：
1. グローバル変数の依存関係マップ
2. 副作用のある関数の一覧
3. 単体テスト不可能な箇所とその理由
4. 安全にリファクタリングできる順序の提案
```

5分後に返ってきた分析レポートの精度に、文字通り椅子から転げ落ちそうになりました。どのグローバル変数が何十箇所で参照されているか、どの関数が外部状態に依存しているか——全部把握した上で、「この順番で直せばリスクが最小になる」というロードマップを出してくれたのです。

> **📝 補足**: Aiderはバージョンによって使えるモデルが異なります。`aider --list-models` で利用可能なモデルを確認してください。

## 😱 3つの痛烈な失敗談

ただ、順風満帆だったわけではありません。最初の2週間で3つの大きな失敗をやらかしました。正直に書きます。

### 失敗1: コンテキスト窓溢れで「AIが忘れる」

最初の大きな壁は、Claudeのコンテキスト制限でした。

在庫計算モジュールが3,000行あり、全部をAiderに読み込ませた状態で作業を続けていたところ、1時間ほど経ったあたりからAIの回答品質が急落し始めました。「さっきそのパターンで直したよね？」と言いたくなるような、前後の文脈を無視した提案が増えてきたのです。

**何が起きていたか**: トークン数が上限に近づき、会話の前半の文脈が切り捨てられ始めていました。「Aiderはコンテキストを賢く管理してくれる」と思い込んでいた私のミスです。

**対策**:
- ファイルを機能単位で分割し、一度に扱うのは500行以内に制限
- セッションをこまめに区切り、新しいセッションでは「前回の決定事項」をプロンプトに明示する
- `/drop` コマンドで不要なファイルをコンテキストから除外する

```bash
# Aider内でファイルを削除してコンテキストを節約
/drop legacy_inventory.py

# 現在のコンテキスト使用量を確認
/tokens
```

### 失敗2: Git管理なしで大量変更して詰む

「とりあえず動いてるコードを直す前にバックアップしとこう」とzipで保存してから作業を始めた私。これが大失敗でした。

Aiderが一度に複数ファイルを変更した結果、ある箇所は新しいコード、ある箇所は古いコードが混在する「壊れた中間状態」に陥りました。zipバックアップに戻すと3時間分の作業が全消え。かといって中間状態のまま進めるとシステムが動かない。

**何が起きていたか**: バージョン管理なしでAIに大量変更をさせた典型的な失敗です。Aiderはデフォルトでgitコミットを自動作成してくれますが、そのgit管理を正しく設定していませんでした。

**対策**: 最初に必ずgitリポジトリを初期化し、Aiderの自動コミット機能を活用する。

```bash
# まずgitリポジトリを初期化（最重要！）
git init
git add .
git commit -m "initial: legacy code before refactoring"

# Aiderを起動（自動コミットが有効になる）
aider --model claude-opus-4-5 --file legacy_inventory.py

# 問題が起きたらすぐロールバック
git log --oneline  # Aiderのコミット履歴確認
git checkout HEAD~3  # 3つ前のコミットに戻る
```

Aiderはリファクタリングのたびに自動でgitコミットを作成してくれます。これが「無限のアンドゥ」として機能します。この安全網があるかないかで、作業の心理的負荷がまったく違います。

### 失敗3: テスト未整備で本番に突撃して障害発生

3週目、調子に乗った私は「コードが綺麗になった！テストは後で書けばいい」とステージング環境すらスキップして本番に直接デプロイしました。

結果: 在庫の入出庫計算で小数点の丸め方が変わっており、夜間バッチで数百件の在庫数値がズレる障害が発生。翌朝の業務開始前に全数補正する羽目になり、クライアントに平謝りでした。

**何が起きていたか**: レガシーコードには「バグっているが、そのバグを前提に動いている部分」が存在します。テストなしでリファクタリングすると、その暗黙の依存関係を壊してしまいます。

**対策**: リファクタリング前にAiderに「現状の振る舞いを記録するテスト」を自動生成させる。

```python
# Aiderへの指示例：既存の振る舞いを記録するテストを生成
# "calculate_stock()関数の現在の振る舞いをすべて網羅するテストを書いてください。
#  バグがあっても、現状の出力をそのまま期待値にしてください（回帰テスト）"

import pytest
from legacy_inventory import calculate_stock

class TestCalculateStockCurrentBehavior:
    """レガシーコードの現状の振る舞いを記録するテスト。
    リファクタリング後もこれらが通ることを確認する。"""

    def test_integer_quantity_calculation(self):
        result = calculate_stock(100, 30)
        assert result == 70  # 現状の実装に基づく期待値

    def test_decimal_rounding_legacy_behavior(self):
        # 注意: この丸め挙動はバグだが、現状のシステムが依存している
        result = calculate_stock(100.5, 30.3)
        assert result == 70.2  # floor丸めではなくtruncate
```

この「現状記録テスト」を最初に書くことで、リファクタリング後の動作確認が格段に楽になりました。

## 🛠️ 実践手順5ステップ

同じことをやりたい方のために、再現可能な手順をまとめました。

### ステップ1: Aiderのセットアップ

```bash
# Aiderをインストール
pip install aider-chat

# ANTHROPIC_API_KEY を設定
export ANTHROPIC_API_KEY="your-api-key-here"

# 動作確認
aider --version
```

### ステップ2: コードベースの分析

いきなり修正を始めてはいけません。まず全体像を把握します。

```bash
# 対象ファイルを指定してAiderを起動
aider --model claude-opus-4-5 \
  --file target_module.py \
  --message "このモジュールの技術的負債を分析し、
             リファクタリング優先順位を付けてください。
             依存関係グラフも含めて教えてください。"
```

### ステップ3: 段階的リファクタリング

一気に全部変えようとしない。小さく確実に進めます。

```bash
# 1ファイルずつ、1機能ずつ進める
aider --model claude-sonnet-4-5 \
  --file utils.py \
  --message "グローバル変数 'GLOBAL_CONFIG' への依存を排除し、
             引数として受け取る形に変更してください。
             変更は最小限にとどめ、既存の呼び出しコードは変更しないでください。"
```

### ステップ4: テスト自動生成

```bash
# リファクタリング後すぐにテストを生成
aider --model claude-opus-4-5 \
  --file utils.py \
  --file tests/test_utils.py \
  --message "utils.pyの全関数に対するpytestテストを
             tests/test_utils.pyに書いてください。
             エッジケースと境界値テストも含めてください。"
```

### ステップ5: レビューと統合

```bash
# 変更のサマリーをAiderに出力させる
aider --model claude-opus-4-5 \
  --file utils.py \
  --message "今回の変更内容をPRの説明文として
             markdown形式でまとめてください。
             変更理由、影響範囲、テスト結果を含めてください。"
```

## ⚙️ Claude API設定とAiderの連携コマンド例

実務で使っているコマンドパターンを公開します。

```bash
# 【基本】単一ファイルのリファクタリング
aider --model claude-opus-4-5 --file legacy.py

# 【分析専用】読み取り専用モードで解析（ファイルを変更しない）
aider --model claude-opus-4-5 \
  --read legacy.py \
  --message "このコードの問題点を箇条書きで列挙してください"

# 【大規模】複数ファイルをまとめて処理（コンテキスト注意）
aider --model claude-opus-4-5 \
  --file module_a.py \
  --file module_b.py \
  --file shared_types.py

# 【コスト節約】分析はsonnet、実装はopusと使い分け
# まずsonnetで方針を決める
aider --model claude-sonnet-4-5 --file legacy.py
# 複雑な実装はopusに切り替え
# Aider内で: /model claude-opus-4-5

# 【設定ファイル】.aider.conf.yml でデフォルト設定
cat > .aider.conf.yml << 'EOF'
model: claude-opus-4-5
auto-commits: true
dirty-commits: true
EOF
```

**APIコスト管理のTips**: 40万行のプロジェクトで2ヶ月使った総コストは約$180でした。モデルの使い分けが重要で、コードの分析・方針決定にはsonnetを使い、実際の実装変更にのみopusを使うことでコストを半分以下に抑えられます。

## 🎯 まとめ

10年物のレガシーシステム刷新という、通常なら半年以上かかるプロジェクトを2ヶ月で完了できた鍵は、Aider×Claudeの活用でした。

ただし、魔法の道具ではありません。私が犯した3つの失敗——コンテキスト管理の軽視、Git管理の省略、テスト軽視——はAIを使っても回避できないヒューマンエラーです。

AIが真価を発揮するのは、**正しいワークフローの中に組み込まれたとき**です。「小さく区切る → git管理 → テスト先行 → AI活用」というサイクルを守れば、Aider×Claudeはレガシーコードを扱うエンジニアの最強のパートナーになります。
